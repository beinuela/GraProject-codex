## 2. 完整版答辩稿

各位老师好，我是 `XXX`，学号是 `XXXXXX`。我的毕业设计题目是《校园物资智能管理系统的设计与实现》。下面我将从课题背景、系统设计、关键实现、工程化改进、项目成果与不足五个方面进行汇报。

首先是课题背景与研究意义。

在高校日常管理中，物资管理是一项基础但非常重要的工作。传统方式通常依赖 Excel、纸质单据或者多个分散系统协同处理，这种方式在实际使用中容易出现信息不同步、审批流程不透明、库存数据滞后、责任追踪困难等问题。尤其是在申领、审批、出入库、调拨和预警处理这几个关键环节中，如果缺少统一的平台支持，就会直接影响管理效率和数据准确性。

基于这样的背景，我设计并实现了一个校园物资智能管理系统。该系统面向校园内部的管理员、仓库管理员、审批人员和部门用户，目标是实现物资管理流程的线上化、规范化和可追踪化，同时兼顾系统的安全性、可维护性和后期扩展能力。

接下来介绍系统的总体设计。

本系统采用前后端分离架构。前端使用 Vue 3、Vite、Element Plus、Pinia 和 ECharts 构建管理界面，主要负责页面展示、交互逻辑和状态管理。后端基于 Spring Boot 3、Spring Security、JWT 和 MyBatis-Plus 实现，主要负责身份认证、权限控制、业务规则处理、事务一致性和数据访问。数据库层支持 MySQL，同时为了本地演示、测试和截图，也提供了基于 H2 的轻量运行模式。

从业务模块上看，系统主要包括以下几个部分。

第一，认证与权限模块。系统支持用户登录、token 刷新、登出以及基于角色的访问控制，为不同角色分配不同的菜单和操作权限。

第二，库存管理模块。包括库存查询、入库、出库、批次管理、仓库与库位管理，是系统的核心数据基础。

第三，申领管理模块。支持部门用户提交申领单，审批人员进行审批，仓库管理员完成出库，最终形成完整的申领闭环。

第四，调拨管理模块。支持仓库之间的物资调拨，包括创建、提交、审批、执行和签收等流程。

第五，智能预警模块。系统能够对低库存、积压、临期、过期以及异常出库等情况进行扫描，并生成预警记录，方便管理人员及时处理。

第六，日志与通知模块。系统记录登录日志、操作日志，并向相关用户推送通知，以支持审计和追踪。

下面介绍系统中的几个关键实现点。

第一个关键点是认证与安全设计。

系统采用 JWT 作为认证方式，前端通过 Bearer Token 调用接口，后端负责 token 的签发、校验和刷新。为了提高安全性，我没有使用源码中的弱默认密钥，而是改为要求通过环境变量提供 JWT 密钥，并在非测试环境启动时强制校验密钥长度，避免因配置不当带来风险。同时，我还为登录、刷新等敏感接口增加了限流机制，防止恶意重试和暴力请求。除此之外，系统在 API 响应层增加了常见安全头，例如 `X-Content-Type-Options`、`X-Frame-Options`、`Referrer-Policy` 和 `Permissions-Policy`，从接口层进一步增强了系统的安全边界。

第二个关键点是库存一致性控制。

在库存业务中，最重要的问题不是“页面能不能展示”，而是“库存数据是否准确”。因此，我把业务规则统一放在后端处理。例如在出库和调拨执行时，系统会优先校验汇总库存，再按批次扣减，并且在同一个事务中完成库存更新、状态流转和日志记录，保证任一环节失败时可以整体回滚，避免出现部分成功、部分失败的数据不一致问题。对于批次扣减，我采用了接近 FEFO，也就是优先出库临近失效批次的策略，使库存管理更符合实际仓储场景。

第三个关键点是业务流程闭环设计。

申领流程不是简单的“提一个单”，而是要覆盖从草稿、提交、审批、出库到签收的完整状态流转。调拨流程同样包含创建、审批、执行、签收等多个阶段。为此，我在服务层明确了各个状态允许的操作，并在关键方法中加入事务控制和必要注释，保证流程可控、逻辑清晰、后期易维护。

第四个关键点是预警与查询优化。

预警模块会对库存、批次和出库记录进行综合扫描，生成不同类型的预警信息。为了避免重复产生相同未处理预警，我在预警记录中加入了去重逻辑，按预警类型、物资、仓库和处理状态进行判断。针对系统中原本容易出现的高频查询问题，我还优化了部分热点逻辑，比如预警扫描中的批量预取、调拨推荐中的仓库信息批量加载，以及申领单出库场景下的关联数据预取，减少了循环中的重复查询，提升了接口执行效率。

下面汇报本系统在工程化方面的改进。

这部分也是我认为本项目相较于普通课程设计更有价值的地方。很多系统只做到“能跑”，但我希望它能更接近真实可交付的软件项目。因此，我围绕文档、安全、测试、性能和监控做了一轮比较完整的工程化整改。

第一，文档方面。

我补充了架构设计文档、开发者贡献指南、API 版本策略、可观测性说明和部署文档，使系统不仅能开发，也能被后续维护者理解和接手。

第二，测试方面。

后端增加了单元测试和基于 Spring Boot Test、MockMvc、H2 的集成测试；前端补充了基于 Vitest 的单元测试，并增加了 Playwright 的关键业务 E2E 测试；性能方面则使用 k6 对登录、库存分页列表、预警列表和操作日志列表进行了冒烟测试。当前本地基线验证结果是：后端 `mvn test` 共 `48` 个测试通过，前端构建通过，前端单测通过，E2E 关键闭环通过。

第三，接口与性能方面。

我将系统中的高增长列表接口统一改造成分页模式，避免原来“全量拉取再前端筛选”的方式导致性能下降和页面卡顿。新的分页结构统一使用 `page`、`size` 请求参数，以及 `records`、`total`、`page`、`size` 响应结构，这也让前后端契约更加清晰。

第四，监控与运维方面。

系统接入了 Spring Boot Actuator 和 Prometheus 指标暴露能力，并结合 Logback JSON 日志、Promtail、Loki 和 Grafana 构建了单机场景下的基础可观测性方案。这样做的意义在于，系统不仅能发现问题，还能定位问题。比如我们可以通过业务指标观察登录失败次数、预警扫描耗时，也可以通过结构化日志和 `requestId` 去追踪某一次具体请求的执行过程。对于异常追踪，系统还预留了 Sentry 的前后端接入能力，在配置 DSN 后可以实现运行时异常上报。

下面介绍系统实现后的效果。

从功能上看，系统已经能够覆盖校园物资管理中的主要环节，包括用户登录、库存查询、申领审批、出入库、调拨处理、预警处理、日志审计和通知提醒，形成了比较完整的业务闭环。

从工程质量上看，系统已经具备一定的可交付能力。它有明确的文档、有基础测试、有安全边界、有分页优化，也有最小可运行的观测方案。相比最初只有功能原型的状态，现在的系统在可靠性、可维护性和后期扩展性方面都有明显提升。

当然，本系统仍然存在一些不足。

第一，目前项目主要面向单机内网部署场景，还没有进一步扩展到分布式架构，也没有引入 Redis 限流、分布式追踪或 Kubernetes 部署方案。

第二，认证模型目前仍采用 JWT 加 localStorage 的方式，虽然已经做了安全加固，但如果未来面向更复杂的生产环境，还可以进一步评估更严格的认证与会话策略。

第三，当前性能测试以本地基线对比为主，能够反映优化方向，但还不能代表跨机器、跨网络环境下的绝对性能表现。

第四，部分智能能力目前仍以规则驱动为主，后续可以在预警推荐、调拨推荐和数据分析方面继续引入更精细的算法优化。

最后，对本次毕业设计做一个总结。

本课题围绕校园物资管理中的实际问题，设计并实现了一套前后端分离的校园物资智能管理系统。系统不仅完成了认证、库存、申领、调拨、预警、日志和通知等核心业务功能，还在安全、测试、性能、文档和监控方面进行了工程化完善。通过本次毕业设计，我进一步加深了对前后端协同开发、业务状态建模、事务一致性、安全设计、测试体系和系统可观测性的理解，也提升了将一个“能运行的系统”打磨为“更接近真实交付系统”的能力。

## 4. 常见问题及参考回答

### 4.1 为什么选择前后端分离架构

参考回答：

我选择前后端分离架构，主要有三个原因。第一，前后端职责更清晰，前端专注交互和展示，后端专注业务规则、事务和数据一致性。第二，这种方式更方便后续扩展，例如未来可以接移动端或其他管理端，而不需要重写业务逻辑。第三，它更符合当前主流开发模式，也更利于测试、部署和维护。

### 4.2 为什么使用 JWT，而不是传统 Session

参考回答：

本项目选择 JWT，主要是考虑到前后端分离场景下接口调用更加直接，前端通过 Bearer Token 即可访问受保护资源。同时为了降低风险，我没有保留弱默认密钥，而是要求通过环境变量提供强密钥，并且增加了刷新 token 轮换、限流和安全头等措施。也就是说，这一轮设计重点不是单纯“用 JWT”，而是把它做成一个相对可控的工程方案。

### 4.3 系统如何保证库存一致性

参考回答：

库存一致性主要通过两层保证。第一层是业务校验，例如出库前先校验汇总库存是否足够。第二层是事务控制，在出库和调拨执行时，批次扣减、汇总库存更新、单据状态流转和日志写入都在同一个事务中完成。这样如果中途出现任何异常，整个过程都会回滚，避免出现库存扣了一半或者状态更新了一半的问题。

### 4.4 为什么要做分页改造

参考回答：

原来的部分列表接口是全量查询，再由前端本地筛选。这种方式在数据量小的时候问题不明显，但随着日志、通知、库存和预警记录增多，接口响应和页面渲染都会变慢，也会增加浏览器内存压力。因此我把高增长列表统一改造成后端分页，前后端接口也统一为 `records`、`total`、`page`、`size` 结构，这样查询和维护都更规范。

### 4.5 预警去重是如何设计的

参考回答：

预警去重的核心目的是避免同一类未处理问题被重复生成。例如同一物资在同一仓库已经存在一条未处理的低库存预警，那么下一次扫描时不应该继续插入相同记录。因此我在预警逻辑中按预警类型、物资、仓库和处理状态做判断，只对未处理状态进行去重。这样既能减少噪声，也能保留真正需要处理的风险信息。

### 4.6 你认为本项目最有价值的改进是什么

参考回答：

我认为最有价值的不是单一功能，而是工程化完善。因为很多课程项目只能做到演示，而我这次重点补齐了文档、安全、测试、分页性能和可观测性，使系统从“能跑”提升到“更接近真实可交付”。例如后端有集成测试，前端有单测和 E2E，安全上有 JWT 校验和限流，运维上有 Prometheus 和日志链路，这些都是后续持续维护的基础。

### 4.7 为什么要引入可观测性，而不是只看控制台日志

参考回答：

只看控制台日志通常适合开发阶段，但不适合长期维护。系统运行后，如果只依赖零散日志，很难快速定位问题。因此我引入了指标、结构化日志和异常追踪三条链路。指标可以看趋势，日志可以定位请求过程，异常追踪可以集中分析错误来源。这样一来，系统不仅能发现问题，还能更快定位问题。

### 4.8 本项目还有哪些改进空间

参考回答：

后续我认为主要有四个方向。第一，向分布式部署扩展，比如引入 Redis 限流和分布式追踪。第二，继续优化认证模型和会话安全策略。第三，进一步完善自动化测试和 CI 流水线。第四，在预警推荐和调拨推荐方面引入更精细的数据分析或算法模型，提高系统的智能化水平。

### 4.10 你这个项目叫校园物资智能管理系统，智能体现在哪里

参考回答：

这个项目中的“智能”，并不是指接入了生成式 AI 或大模型对话，而是指系统具备了基于业务规则和数据分析的自动感知、自动判断和辅助决策能力。它的智能主要体现在四个方面。

第一，智能预警。系统能够自动扫描库存和业务数据，识别低库存、积压、临期、过期和异常出库等风险，并生成预警记录。这相当于把原来依赖人工排查的问题，变成了系统自动发现问题。

第二，智能推荐。在调拨业务中，系统不是只让用户手工填写，而是会结合不同仓库的库存情况，给出调拨建议。这体现的是一种基于当前库存分布和业务需求的辅助决策能力。

第三，智能批次处理。在出库和调拨执行时，系统会按照批次规则自动选择更合适的库存批次，例如优先处理更接近失效日期的批次。这种自动化决策减少了人工判断成本，也更符合实际仓储管理逻辑。

第四，智能分析与可视化支撑。系统通过仪表盘、大屏和统计分析模块，对库存、预警、日志和业务趋势进行汇总展示，帮助管理人员更快发现异常和做出判断。这种从数据中提取管理信息的能力，也是“智能管理”的一种体现。

如果老师继续追问“这是不是严格意义上的人工智能”，可以这样回答：严格来说，这个项目当前更偏向规则驱动和数据驱动的智能管理系统，而不是基于机器学习的大模型系统。我在命名中使用“智能”，主要是因为系统已经具备了自动预警、自动推荐和辅助决策能力。后续如果继续扩展，可以进一步引入更精细的预测模型和推荐算法，让“智能”从规则型走向模型型。

## 5. 答辩时的表达建议

- 开场先说“系统解决了什么问题”，不要一上来就背技术栈。
- 讲功能时强调“业务闭环”，讲技术时强调“为什么这样做”。
- 老师如果追问实现细节，优先从事务、安全、性能和测试四个角度回答。
- 如果遇到不会的问题，不要硬答，可以先说明当前实现范围，再说后续改进方向。
