# 毕业答辩项目演示完整流程

本文档用于《校园物资智能管理系统设计与实现》答辩现场系统演示。内容基于当前仓库的前端路由、后端接口、数据库脚本、测试用例和既有答辩稿整理，目标是保证演示过程顺畅、功能闭环清晰、时间可控。

### 不建议重点演示模块

| 内容 | 原因 | 现场说法 |
| --- | --- | --- |
| 独立配送任务页面 | 当前仓库没有独立配送任务实体、路由或接口 | “当前实现将配送交付简化到出库、调拨执行和签收状态中，重点演示已实现的出库与签收闭环。” |
| 采购员角色 | 数据库与菜单中没有 `PURCHASER` 角色 | “采购来源在入库管理中用来源类型记录，未单独拆分采购员角色。” |
| 调度员角色 | 数据库与菜单中没有 `DISPATCHER` 角色 | “调拨执行由仓库管理员完成，审批由审批人员完成。” |
| 大规模机器学习模型 | 代码中没有机器学习训练或模型服务 | “智能部分以规则预警、需求趋势、补货建议和可视化辅助决策为主。” |

## 1. 演示前准备

### 数据准备清单

优先使用仓库已有脚本：

```powershell
mysql -u root -p campus_material < sql/schema.sql
mysql -u root -p campus_material < sql/seed.sql
```

答辩前确认以下数据存在：

- 用户角色：`ADMIN`、`WAREHOUSE_ADMIN`、`DEPT_USER`、`APPROVER`
- 演示账号：`admin`、`warehouse`、`dept`、`approver`
- 部门：学校本部、后勤处、保卫处、医务室、计算机学院
- 物资分类：防疫类、医疗急救类、食品饮水类、照明通信类、防汛防灾类
- 物资信息：医用口罩、84消毒液、急救包、瓶装饮用水、强光手电
- 仓库：科学校区总仓、东风校区分仓、医务室物资仓
- 库存与批次：至少有可出库批次、临期批次和低库存样例
- 申领单：至少有一条待审批或可新建的申领单
- 调拨单：至少有一条可执行或可新建的调拨单
- 预警记录：至少有一条未处理预警
- 日志与通知：已有登录日志、操作日志、通知消息

## 2. 推荐演示账号与角色切换

### 实际角色规划

| 账号 | 角色编码 | 角色名称 | 适合演示功能 |
| --- | --- | --- | --- |
| `admin` | `ADMIN` | 系统管理员 | 登录、系统首页、用户管理、角色管理、部门管理、基础数据、日志、安全策略、统计分析 |
| `warehouse` | `WAREHOUSE_ADMIN` | 仓库管理员 | 物资信息、供应商、仓库、库位、库存查询、入库、出库、调拨执行、预警处理、统计分析 |
| `approver` | `APPROVER` | 审批人员 | 申领审批、调拨审批、预警查看、统计分析 |
| `dept` | `DEPT_USER` | 部门用户 | 新建申领、提交申领、签收、查看统计分析 |

### 是否需要切换账号

推荐现场不要频繁切换账号。可以采用下面两种方案：

1. 稳定方案：主要使用 `admin` 演示系统整体功能，再说明不同角色的菜单边界。适合 5 分钟演示。
2. 完整方案：使用 `dept -> approver -> warehouse -> dept` 演示一条申领闭环，再切回 `admin` 总结统计和日志。适合 8 分钟演示。

如果时间紧，建议先保证 `admin` 登录后能展示完整页面，再选择一条已有申领单演示审批、出库或签收中的关键节点。

## 4. 逐步演示流程

### 步骤一：登录系统

演示页面：

登录页。

操作：

1. 打开 `http://localhost:5173/login`。
2. 输入管理员账号 `admin` 和演示密码。
3. 点击“登录系统”。
4. 进入系统首页。

讲解话术：

“首先进入系统登录页面。系统采用前后端分离方式，登录成功后后端签发 JWT，前端在后续请求中通过 `Authorization: Bearer Token` 访问受保护接口。系统会根据用户角色加载对应菜单，实现不同岗位的操作边界控制。”

体现功能：

- 用户认证
- JWT 访问令牌
- 角色菜单加载
- 前后端分离接口调用

备用说明：

“如果现场登录接口响应较慢，可以说明本地已经完成登录状态保持和权限菜单加载，后续页面均通过同一认证机制访问。”

### 步骤二：查看系统首页与数据看板

演示页面：

系统首页 `/dashboard`。

操作：

1. 查看顶部标题“系统首页”。
2. 查看物资种类、仓库数量、库存记录、未处理预警等指标。
3. 查看库存占比分布和近六个月出入库趋势。
4. 点击“进入分析中心”后再返回首页。

讲解话术：

“系统首页用于汇总管理人员最关心的数据，包括物资、仓库、库存和未处理预警数量。下方图表来自后端统计接口，可以辅助管理人员快速了解当前库存结构和出入库趋势。”

体现功能：

- 首页运行概览
- 库存与预警指标聚合
- ECharts 图表展示
- 后端统计接口联动

备用说明：

“如果图表数据较少，说明当前是演示数据集，真实使用时随着入库、出库、申领和调拨记录增加，图表会自动反映业务变化。”

### 步骤三：展示角色权限与基础数据

演示页面：

用户管理、角色管理、部门管理、物资分类、物资信息、仓库管理。

操作：

1. 进入“用户管理”，展示账号、角色、部门、状态。
2. 进入“角色管理”，展示 `ADMIN`、`WAREHOUSE_ADMIN`、`DEPT_USER`、`APPROVER`。
3. 进入“物资信息”，展示物资编码、规格、单位、安全库存、有效期等字段。
4. 进入“仓库管理”，展示不同校区仓库。

讲解话术：

“这一部分是业务流程运行的基础。用户和角色决定谁能访问哪些页面，物资和仓库数据决定库存管理的对象。物资档案中配置了安全库存和保质期信息，后续低库存预警和临期预警都依赖这些基础数据。”

体现功能：

- RBAC 基础数据
- 部门与用户归属
- 物资档案维护
- 仓库与校区基础信息

备用说明：

“为了现场演示稳定，基础数据建议提前准备好，现场只展示结构，不建议临时删除或大量修改。”

### 步骤四：查看库存与批次

演示页面：

库存查询 `/inventory/list`。

操作：

1. 点击“库存查询”。
2. 查看物资、仓库、当前库存、锁定库存、安全库存。
3. 使用仓库或物资筛选。
4. 点击批次相关按钮或查看批次列表，展示批次号、余量、生产日期和有效期。

讲解话术：

“库存查询展示的是物资在仓库维度的汇总库存，同时系统保留批次数据。出库和调拨时，后端会按照有效期优先的规则扣减批次，避免人工随意选择导致临期物资积压。”

体现功能：

- 库存分页查询
- 批次库存管理
- 锁定库存展示
- 安全库存对比

备用说明：

“如果某个筛选条件下没有数据，可以清空筛选条件，使用已有的科学校区总仓或医务室物资仓数据继续演示。”

### 步骤五：演示入库管理

演示页面：

入库管理 `/inventory/stock-in`。

操作：

1. 点击“入库管理”。
2. 点击“新建入库”。
3. 选择仓库，例如“科学校区总仓”。
4. 选择物资，例如“医用口罩”或“84消毒液”。
5. 输入批次号、数量、生产日期、有效期和备注。
6. 点击保存。
7. 返回库存查询，观察库存数量增加。

讲解话术：

“入库操作会同时写入入库单、入库明细、批次库存和汇总库存。这样既能查到历史入库记录，也能按批次管理当前库存。”

体现功能：

- 入库单登记
- 批次生成
- 汇总库存同步
- 操作日志记录

备用说明：

“如果不想现场新增数据，可以直接打开已有入库记录，说明入库后会同步影响库存查询和统计图表。”

### 步骤六：演示申领创建与提交

演示页面：

申领审批 `/apply/list`。

使用角色：

建议使用 `dept`，时间紧可用 `admin` 说明流程。

操作：

1. 切换到部门用户 `dept`。
2. 进入“申领审批”。
3. 点击“新建申领”。
4. 选择部门、紧急程度、申领原因和使用场景。
5. 选择物资，例如“医用口罩”或“急救包”，输入数量。
6. 保存后点击“提交”。

讲解话术：

“部门用户可以根据实际需求创建申领单。提交时，后端会检查库存是否能够满足本次申领，并锁定对应库存，避免多个申请同时提交时出现超领问题。”

体现功能：

- 申领单创建
- 申领明细
- 库存校验
- 锁定库存
- 状态从 `DRAFT` 到 `SUBMITTED`

备用说明：

“如果现场新建时提示库存不足，可以减少申领数量，或者选择库存较充足的医用口罩。”

### 步骤七：演示审批处理

演示页面：

申领审批 `/apply/list`。

使用角色：

`approver` 或 `admin`。

操作：

1. 切换到审批人员 `approver`。
2. 打开“申领审批”。
3. 找到状态为 `SUBMITTED` 的申领单。
4. 点击“通过”或“驳回”。
5. 输入审批备注。

讲解话术：

“审批人员只能处理已提交的单据。审批通过后，单据状态变为 `APPROVED`，后续由仓库管理员根据审批结果出库；如果驳回，系统会释放之前锁定的库存。”

体现功能：

- 角色权限控制
- 状态机控制
- 审批备注
- 库存锁定释放

备用说明：

“如果没有待审批单据，可以使用脚本初始化的医务室常规补充申请，或提前准备一条 `SUBMITTED` 状态数据。”

### 步骤八：演示出库与签收

演示页面：

出库管理 `/inventory/stock-out` 和申领审批 `/apply/list`。

使用角色：

出库使用 `warehouse`，签收使用 `dept`。

操作：

1. 使用 `warehouse` 登录。
2. 进入“出库管理”。
3. 点击“新建出库”。
4. 选择审批通过的申领单、仓库和物资数量。
5. 保存出库。
6. 回到“申领审批”，确认单据状态变为 `OUTBOUND`。
7. 使用 `dept` 登录，点击“签收”，状态变为 `RECEIVED`。

讲解话术：

“出库环节会扣减批次库存和汇总库存，并回写申领明细的实发数量。签收完成后，申领流程从申请、审批、出库到接收形成闭环。”

体现功能：

- 出库单登记
- 批次扣减
- 汇总库存扣减
- 申领状态闭环
- 操作日志审计

备用说明：

“当前系统没有单独拆分配送任务表，演示中以出库和签收表示物资交付完成。论文中如果描述了更细的配送环节，可以说明这是后续可扩展方向。”

### 步骤九：演示调拨管理

演示页面：

调拨管理 `/transfer/list`。

使用角色：

`warehouse` 创建和执行，`approver` 审批。

操作：

1. 使用 `warehouse` 进入“调拨管理”。
2. 点击“新建调拨”。
3. 选择调入仓库、调出仓库、物资和数量。
4. 可点击推荐调出仓，查看系统根据库存和校区距离给出的候选仓库。
5. 提交调拨单。
6. 使用 `approver` 审批。
7. 使用 `warehouse` 执行调拨并签收。

讲解话术：

“调拨用于解决不同校区或仓库之间库存不均衡的问题。系统提供调拨推荐，结合仓库库存和校区距离进行排序；执行调拨时，后端在同一事务中完成来源仓扣减、目标仓增加和批次搬移。”

体现功能：

- 调拨单创建
- 调拨审批
- 调拨推荐
- 跨仓库存移动
- 批次镜像生成
- 调拨签收

备用说明：

“如果推荐结果为空，通常是因为候选仓库库存不足或排除了目标仓库，可以减少数量或更换物资。”

### 步骤十：演示预警管理

演示页面：

预警管理 `/warning/list`。

操作：

1. 打开“预警管理”。
2. 查看未处理预警，包括低库存、库存积压、临期、过期、异常出库。
3. 点击“扫描预警”。
4. 选择一条预警点击“处理”，填写处理备注。

讲解话术：

## 5. 重点业务闭环演示

推荐主线：

部门用户提交物资申领需求 -> 审批人员审批 -> 系统锁定库存 -> 仓库管理员出库 -> 部门用户签收 -> 系统首页和统计分析反映业务变化 -> 库存不足或临期时生成预警。

| 环节 | 页面 | 角色 | 数据准备 | 成功结果 |
| --- | --- | --- | --- | --- |
| 创建申领 | 申领审批 | `dept` | 选择有库存物资，建议医用口罩或急救包 | 新增 `DRAFT` 单据 |
| 提交申领 | 申领审批 | `dept` | 数量不超过可用库存 | 状态变为 `SUBMITTED`，库存出现锁定量 |
| 审批通过 | 申领审批 | `approver` | 有待审批单据 | 状态变为 `APPROVED` |
| 出库 | 出库管理 | `warehouse` | 选择审批通过的申领单和锁定仓库 | 生成出库单，库存扣减，状态变为 `OUTBOUND` |
| 签收 | 申领审批 | `dept` | 已出库申领单 | 状态变为 `RECEIVED` |
| 查看结果 | 系统首页、统计分析、日志 | `admin` | 业务操作已完成 | 指标、图表、操作日志更新 |
| 预警处理 | 预警管理 | `warehouse` 或 `approver` | 有未处理预警，或点击扫描 | 预警状态变为 `HANDLED` |

注意：

- 当前系统没有独立配送任务模块，不要把“出库到签收”讲成完整物流配送系统。
- 当前“智能”不要讲成机器学习平台，应讲为规则预警、趋势分析、补货建议和调拨推荐。
- 如果现场切换账号来不及，可以使用管理员账号演示页面和已有数据，再口头说明角色边界。

各位老师好，下面我演示《校园物资智能管理系统设计与实现》的主要功能。这个系统主要面向校园物资管理场景，解决传统方式中库存查询不及时、申领审批不透明、出入库难追踪和预警依赖人工排查的问题。

### 详细版口播稿，适合 8 分钟演示

各位老师好，下面我进行项目系统演示。我的毕业设计题目是《校园物资智能管理系统设计与实现》。这个系统围绕校园内部物资管理设计，重点解决传统 Excel 或纸质单据方式下的数据分散、库存更新不及时、审批流程不透明和责任追踪困难等问题。

首先演示登录。系统使用 Vue 3 前端和 Spring Boot 后端，登录时前端调用 `/api/auth/login`，后端校验账号和密码后返回 JWT。后续接口请求都会携带 Token，后端通过 Spring Security 进行认证和授权。

登录成功后进入系统首页。这里展示的是系统运行概览，包括物资种类、仓库数量、库存记录和未处理预警。页面下方还有库存占比和出入库趋势图，数据来自后端统计接口。通过这个页面，管理人员可以快速掌握当前物资和库存状态。

接下来展示权限和基础数据。当前项目实际存在四类角色：系统管理员、仓库管理员、审批人员和部门用户。系统管理员可以维护用户、角色、部门、校区、物资分类、物资信息和仓库信息。物资档案里有安全库存和保质期字段，这些字段会直接参与库存预警、临期预警和补货建议。

然后进入库存查询。系统既记录物资在仓库维度的汇总库存，也记录批次库存。批次中包含批次号、入库数量、剩余数量、生产日期和有效期。后端出库和调拨时会按照有效期优先的规则扣减批次，减少临期物资长期积压。

入库管理用于登记采购或补充入库。新增入库时选择仓库、物资、数量、批次号、生产日期和有效期。保存后系统会同时写入入库单、入库明细、库存汇总和批次库存，保证后续库存查询和统计分析都能看到变化。

接下来演示申领闭环。部门用户可以创建申领单，填写申领原因、使用场景、紧急程度和物资明细。提交时后端会检查是否有一个仓库可以满足本次申领，并锁定对应库存，防止多个申请同时占用同一批库存。

审批人员登录后可以查看已提交的申领单，并进行通过或驳回。只有 `SUBMITTED` 状态允许审批，审批通过后状态变为 `APPROVED`；如果驳回，系统会释放提交时锁定的库存。这样可以避免单据状态被随意跳转。

审批通过后由仓库管理员执行出库。出库时系统会校验库存、扣减批次、更新汇总库存，并回写申领明细中的实发数量。出库完成后申领单状态变为 `OUTBOUND`，部门用户签收后变为 `RECEIVED`，至此完成从申请到签收的业务闭环。

调拨管理用于处理跨仓库物资补充。仓库管理员创建调拨单，选择调出仓、调入仓和物资数量；审批人员通过后，仓库管理员执行调拨。执行时后端会在同一事务中扣减来源仓库存、增加目标仓库存，并生成目标仓批次记录。系统还提供调拨推荐，按库存和校区距离给出候选仓库。

预警管理用于自动发现风险。系统通过定时任务和手动扫描识别低库存、库存积压、临期、过期和异常出库。对于相同类型、相同物资、相同仓库且未处理的预警，系统会做去重，避免重复提醒。管理人员处理预警后可以留下备注，形成处理记录。

通知和日志部分用于追踪过程。通知中心展示预警、审批和事件消息；操作日志记录申领、出库、调拨、预警处理等关键操作；登录日志记录用户登录情况。这些功能让系统具备基本的审计能力。

最后展示统计分析。统计分析页面使用 ECharts 展示库存占比、仓库分布、出入库趋势、部门申领排行、有效期统计、应急消耗趋势、需求预测和补货建议。这里的“智能”主要体现为规则预警、历史数据趋势、补货建议和可视化辅助决策，不是大型 AI 模型。

从技术上看，前端使用 Vue 3、Vite、Element Plus、Pinia 和 ECharts；后端使用 Spring Boot 3、Spring Security、JWT、MyBatis-Plus 和 MySQL。系统还补充了后端测试、前端构建与 E2E 测试、分页接口、安全头、限流和基础可观测性。整体来看，本项目完成了校园物资从基础数据、库存、申领、审批、出库、调拨、预警到统计分析的主要闭环。我的演示完毕，谢谢各位老师。

## 8. 老师可能追问的问题与回答

### 1. 为什么要做这个系统？

校园物资管理涉及多个部门、多个仓库和多种物资，如果只靠表格或人工记录，容易出现库存不准、审批不透明和追踪困难。本系统把物资档案、库存、申领、审批、出入库、调拨和预警统一到一个平台中，提高流程规范性和数据可追踪性。

### 2. 系统相比普通库存管理系统有什么特点？

普通库存系统往往只关注入库、出库和库存数量。本系统结合校园场景加入了申领审批、跨仓调拨、角色权限、预警处理、通知日志和统计分析，重点是形成管理流程闭环。

### 3. “智能”体现在哪里？

当前智能主要体现在规则和数据统计驱动的辅助管理，包括低库存预警、库存积压预警、临期和过期预警、异常出库识别、调拨推荐、需求趋势分析和补货建议。它不是生成式 AI 或大型机器学习平台。

### 4. 权限控制是怎么实现的？

系统使用 Spring Security 开启方法级权限控制，Controller 上通过 `@PreAuthorize` 限制不同角色的访问。登录后菜单也会根据角色返回，例如管理员菜单最完整，部门用户主要能使用申领和统计功能。

### 5. JWT 登录认证流程是什么？

用户登录时提交用户名和密码，后端校验成功后生成 access token 和 refresh token。前端把 access token 放入请求头，后端过滤器解析并校验 token，然后把用户身份和角色放入安全上下文。

### 6. 数据库有哪些核心表？

核心表包括用户角色部门表、物资分类和物资信息表、仓库和库位表、库存汇总表、批次库存表、入库和出库表、申领主从表、调拨主从表、预警记录表、通知表、登录日志和操作日志表。

### 7. 库存预警是怎么触发的？

预警可以由定时任务触发，也可以在页面手动扫描。系统会检查库存是否低于物资档案中的安全库存，是否超过安全库存多倍形成积压，批次是否临期或过期，以及近期出库量是否异常。

### 8. 申领审批流程如何保证闭环？

后端定义了明确状态：`DRAFT`、`SUBMITTED`、`APPROVED`、`REJECTED`、`OUTBOUND`、`RECEIVED`。每个接口只允许在特定状态执行，出库和签收会推动状态继续流转，最终形成闭环。

### 9. 如果库存不足怎么办？

申领提交和出库时都会检查库存。如果不能满足申请，接口会返回业务错误，并且不会更新状态。出库库存不足时还会生成低库存预警，提醒仓库管理员补货或调拨。

### 10. 前后端如何交互？

前端通过 Axios 调用后端 REST API，统一处理响应码、错误提示和 Token。后端统一返回 `ApiResponse`，分页接口返回 `PageResult`，结构包括 `records`、`total`、`page`、`size`。

### 11. ECharts 图表数据从哪里来？

图表数据来自后端统计接口，例如库存占比、仓库分布、出入库趋势、部门申领排行和有效期统计。后端通过 MySQL 查询聚合业务表，再返回给前端渲染。

### 12. 系统测试做了哪些？

已有测试包括后端单元测试、Spring Boot 集成测试、前端 Vitest 单元测试、Playwright E2E 测试和 k6 冒烟脚本。集成测试覆盖登录刷新、申领出库闭环、调拨执行、预警处理、分页接口和权限校验。

### 13. 项目还有哪些不足？

当前系统主要面向单机内网部署，没有实现分布式架构；独立配送任务模块尚未拆分；智能能力以规则和统计为主，还不是复杂模型；移动端和更细粒度权限也可以继续完善。

### 14. 后续可以怎么优化？

后续可以引入 Redis 做缓存和分布式限流，完善配送任务和移动端签收，增加更细的权限配置，引入更精细的预测模型，补充 CI/CD 流水线和更完整的生产监控。

### 15. 为什么选择 Vue + Spring Boot + MySQL？

Vue 3 适合快速构建前后端分离管理页面，Element Plus 提供稳定的后台组件；Spring Boot 生态成熟，适合实现认证、事务和业务接口；MySQL 适合结构化业务数据存储，也便于本地部署和答辩演示。

| 页面数据为空 | 演示数据未准备或筛选条件过窄 | 清空筛选条件，使用已有数据 | 导入 `sql/seed.sql` | “当前显示取决于数据库业务数据。” |
| 图表不显示 | 统计数据为空或接口异常 | 刷新页面，打开统计分析备用截图 | 提前准备几条出入库和申领数据 | “图表基于业务数据聚合，演示库数据少时趋势较平。” |
| 端口占用 | 旧 Java 或 Node 进程未关闭 | `netstat -ano | findstr :8080`，再按 PID 结束 | 答辩前只保留一个后端实例 | “端口占用是本地进程冲突，清理后即可启动。” |
| 账号权限不对 | 使用了普通用户访问管理功能 | 切换 `admin` 或对应角色 | 账号贴在演示卡片上 | “系统按角色控制菜单和接口权限。” |
| 申领流程卡住 | 申领数量超过库存、状态不允许操作 | 减少数量或使用已有待审批单据 | 准备一条待审批单据 | “后端状态机限制非法流转，这是业务保护。” |
| 调拨推荐为空 | 候选仓库存不足或排除目标仓 | 减少数量，更换物资 | 使用医用口罩作为推荐物资 | “推荐结果依赖库存和校区信息。” |
| 预警扫描被限流 | 连续点击高风险接口 | 等待后继续，或使用已有预警 | 现场只点一次扫描 | “这是安全限流机制，防止重复高风险操作。” |
| 浏览器异常 | 缓存或登录状态过期 | 清理 localStorage 或重新登录 | 提前开无痕窗口备用 | “重新登录后 Token 会重新签发。” |
