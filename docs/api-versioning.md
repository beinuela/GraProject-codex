# API 版本控制策略

## 1. 当前基线

- 当前未版本化的 `/api/**` 视为内部 `v1` 基线
- 本轮不批量迁移到 `/api/v1/**`
- 原因：前后端主要消费者都在本仓库内，现阶段整体改 URI 的收益低于改造成本

## 2. 何时进入新版本

当接口发生以下“破坏性变更”时，才创建新主版本：

- 删除字段
- 修改字段语义且无法兼容旧消费者
- 修改返回结构主形态
- 调整权限模型导致旧调用方无法继续使用
- 参数含义变化，且无法通过新增字段保持兼容

## 3. 新版本规则

- 新主版本统一采用 URI 版本化：`/api/v2/**`
- 旧版本至少保留一个发布周期
- 旧接口在 OpenAPI 与文档中标记 `deprecated`
- 新旧版本并行期间，前端必须显式声明切换窗口和回滚策略

## 4. 非破坏性变更处理

以下变更不创建新主版本：

- 新增可选字段
- 新增可选查询参数
- 新增只读接口
- 在不破坏现有调用方的前提下增加返回信息
- 将裸数组升级为分页对象，但前后端在同一仓库中同步改造且同轮上线

## 5. 变更评审检查项

在创建新版本前，请回答以下问题：

1. 是否真的无法通过向后兼容方式解决？
2. 是否已有消费者依赖旧结构？
3. 是否已经给出迁移窗口、回滚方案和废弃时间点？
4. 是否同步更新了 `README`、接口文档和前端适配说明？

## 6. 本项目当前约束

- 本轮保留 `/api/**` 路径不变
- 分页、公共错误码、安全头和限流策略都作为 `v1` 内部增强
- 下一次出现破坏性变更时，再统一进入 `/api/v2/**`
