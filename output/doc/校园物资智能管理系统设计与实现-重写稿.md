# 校园物资智能管理系统设计与实现

## 摘要

针对高校物资管理中台账分散、审批流程不清晰、库存批次追踪困难、预警处置依赖人工经验以及管理信息难以统一汇总等问题，本文结合校园物资保障工作的实际需求，设计并实现了一套基于 B/S 架构的校园物资智能管理系统。系统采用前后端分离模式，前端基于 Vue 3、Vite、Pinia、Element Plus 和 ECharts 构建业务页面与统计图表，后端基于 Spring Boot 3、Spring Security、JWT、MyBatis-Plus 和 MySQL 实现认证授权、业务处理、事务控制与数据存储，并提供 H2 运行模式以支持截图展示和自动化测试。系统围绕物资档案维护、库存与批次管理、部门申领、审批处理、出库签收、仓间调拨、预警识别、通知提醒、日志审计和统计分析等业务环节进行设计，形成了覆盖多角色、多单据和多状态流转的完整业务链路。为提升系统智能化水平，本文将预警识别、补货建议、趋势分析、候选仓排序与 DeepSeek 大模型辅助分析进行整合，系统能够围绕预警记录关联的库存数据、批次信息、出库波动和待处理业务上下文生成风险等级、原因摘要、处置建议、责任角色和建议时限等结构化结果，并通过任务留痕与调用日志保证分析过程可追溯；当模型服务异常时，系统可自动回退到规则分析结果，从而保持功能稳定性与业务连续性。测试结果表明，系统核心业务流程运行稳定，认证、库存、申领、调拨、预警和统计等功能能够完成有效联动，具备较好的信息化、规范化和辅助决策支持能力，可为高校物资管理数字化建设提供参考。

**关键词：**校园物资管理；B/S 架构；Spring Boot；Vue 3；DeepSeek；辅助决策

## ABSTRACT

This paper designs and implements a campus material intelligent management system for problems such as scattered ledgers, unclear approval procedures, weak batch traceability, heavy reliance on manual judgment in warning handling, and insufficient integration of management information in campus material management. The system adopts a browser/server architecture and a front-end/back-end separation pattern. The front end is built with Vue 3, Vite, Pinia, Element Plus, and ECharts, while the back end is implemented with Spring Boot 3, Spring Security, JWT, MyBatis-Plus, and MySQL. An H2-based runtime mode is also provided for automated testing and demonstration. Around the main business links of master data maintenance, inventory and batch management, department requisition, approval processing, outbound execution, warehouse transfer, warning identification, notification, audit logging, and statistical analysis, the system forms an integrated management workflow covering multiple roles, documents, and state transitions. To strengthen intelligent support, the system combines warning identification, replenishment suggestion, trend analysis, candidate warehouse ranking, and DeepSeek-based warning analysis. It can analyze warning-related inventory data, batch information, outbound fluctuations, and pending business context, then generate structured results including risk level, cause summary, handling suggestions, responsible role, and suggested deadline. Task records and model call logs are used to ensure traceability, and the system automatically falls back to rule-based analysis when the model service is unavailable or abnormal. Test results show that the system can stably support the coordination of authentication, inventory, requisition, transfer, warning, and statistical functions, and it has practical value for the digital, standardized, and intelligent management of campus materials.

**Keywords:** campus material management; B/S architecture; Spring Boot; Vue 3; DeepSeek; decision support

## 1 绪论

### 1.1 项目背景及意义

随着智慧校园建设持续推进，高校内部各类保障业务逐步由线下管理转向线上协同，物资管理也由简单的仓储台账维护演变为覆盖采购入库、库存控制、部门申领、审批处理、出库签收、仓间调拨、风险预警和统计分析的综合性业务场景。校园物资种类多、使用部门分散、流转角色复杂，不同类型物资在保质期、批次管理、安全库存和责任追溯方面存在明显差异。如果仍以 Excel、纸质登记或多个分散页面协同处理，不仅容易造成库存信息更新不及时、审批状态难跟踪、批次流向不明确和风险处置滞后，还会增加管理人员重复沟通和人工核对的成本。

近年来，高校后勤、实验室管理以及医疗机构物资流通等研究表明，现代物资管理系统不应停留在“能记账、能查库存”的层面，而应具备流程协同、实时更新、异常预警和辅助决策等能力。对于校园场景而言，物资管理既关系到日常行政运行，也关系到应急保障和公共服务质量。当物资需求集中变化、某类物资临近失效或跨仓资源分布不均时，如果缺少统一平台支撑，就难以及时做出准确判断。因此，将认证授权、库存批次、申领审批、调拨执行、预警识别和统计分析整合到一个系统中，具有较强的现实意义。

从工程实现角度看，前后端分离、JWT 认证、RBAC 权限控制、事务一致性、自动化测试和可观测性已经成为管理信息系统开发中的重要组成部分。毕业设计如果仅实现页面展示，而没有把核心业务规则、安全边界和测试验证落到代码中，论文就容易停留在概念描述层面。基于此，本文选择以真实仓库代码为基础，围绕校园物资管理业务构建一套完整系统，并把库存一致性、流程闭环、日志审计、预警分析和智能辅助决策作为重点内容展开论述。

本文的研究意义主要体现在三个方面。第一，在业务管理层面，系统通过统一物资档案、仓库、库存、申领、调拨、预警和日志数据，能够提升校园物资流转过程的透明度、规范性和可追溯性。第二，在技术实现层面，系统通过 Spring Boot 与 Vue 的前后端分离架构，将认证授权、分页治理、事务控制、规则预警和大模型分析进行组合，实现了一个更接近真实交付场景的软件系统。第三，在智能化探索层面，系统并未把大模型作为独立聊天工具，而是将 DeepSeek 与预警处置、风险研判和辅助建议生成过程结合起来，使“智能管理”建立在真实业务数据和审计留痕基础之上，从而增强系统的实用性与扩展性。

### 1.2 国内外研究现状

国内有关高校后勤与物资管理的研究较多聚焦于物资台账、库存监控、出入库流程和智慧校园协同等方面。唐锡雷围绕高职院校后勤物资管理系统进行了设计与实现研究，说明了校园后勤物资业务具有多角色、多环节和高协同的特点[1]。李晓英、高菲从智慧校园建设角度讨论了高职院校物资管理系统创新方向，指出校园物资平台需要兼顾业务规范化和信息共享[2]。曾文勇、刘小玲关于“数智赋能物资供应管理”的报道，则进一步体现出物资供应管理正在从传统记录型系统走向数据驱动和智能辅助阶段[3]。张俊萌、张华贵对医疗机构物资流通系统的研究表明，库存流转、审批处理、预警提示和数据跟踪已经成为此类系统的共性要求[4]。厉志安等人围绕高校实验室危化品溯源管理的研究，强调了追溯、预警和智能管理对于高风险物资场景的重要性[5]。

在技术路线方面，相关研究已经围绕 JWT 与 RBAC 权限控制、业务逻辑安全、Vue 与 Spring Boot 的前后端分离开发模式、动态路由组织、可视化页面构建以及基于 Spring 的典型管理信息系统实现展开了较为充分的讨论，可为本文系统在技术选型、模块组织、安全设计和工程实现方面提供参考[6-20]。

国外和英文文献除关注全栈开发与系统实现外，也从网络环境下的用户行为和信息系统交互效率角度为系统界面组织与功能设计提供了补充视角[21]。这类研究有助于本文在界面组织、流程引导和用户交互体验方面形成更合理的实现思路。

不过，从现有研究和公开实现来看，仍存在三个值得改进的方向。其一，很多校园类物资系统重视基础台账和常规查询，但对库存批次、状态流转、跨仓调拨和审计追踪的结合不够充分。其二，部分系统把“智能”表述停留在统计展示或规则提醒层面，缺少与业务上下文深度结合的分析能力。其三，大模型能力在管理系统中的接入经常被写成通用问答或独立智能助手，缺少与具体业务流程、结构化输出和调用留痕的结合。基于这些不足，本文尝试构建一套面向校园物资管理的完整业务系统，并把 DeepSeek 辅助分析嵌入预警处置流程中，以增强系统的研判和决策支持能力。

### 1.3 研究内容

本文以校园物资管理系统为研究对象，围绕真实项目代码和数据库设计展开，主要研究内容包括以下几个方面。

第一，结合高校物资管理场景梳理业务角色与流程边界，明确系统需要覆盖的核心业务，包括基础资料维护、库存与批次管理、部门申领、审批处理、出库签收、仓间调拨、预警处置、通知日志和统计分析。

第二，基于前后端分离模式完成系统整体技术路线设计。前端通过 Vue 3、Vite、Pinia、Element Plus 和 ECharts 负责页面交互与数据展示；后端通过 Spring Boot 3、Spring Security、JWT、MyBatis-Plus 和 MySQL 承担认证授权、业务处理、事务控制和数据持久化。

第三，围绕库存一致性和业务闭环设计关键实现机制。本文重点分析申领审批、库存锁定、出库回写、跨仓调拨、批次优先扣减和状态流转控制等核心逻辑，确保系统不仅“可用”，而且“正确”。

第四，设计并实现系统智能能力。本文将低库存、积压、临期、过期和异常出库识别、补货建议、趋势分析、候选仓排序与 DeepSeek 大模型分析结合起来，形成预警识别、风险说明和辅助建议一体化的智能管理能力。

第五，通过自动化测试、前端构建验证、端到端流程验证和本地性能基线对系统进行验证，说明系统功能是否完整、业务是否闭环，以及设计是否具有可实现性与稳定性。

## 2 技术选型

### 2.1 开发框架选型

本系统采用前后端分离开发模式。前端选择 Vue 3、Vite、Pinia 和 Element Plus，后端选择 Spring Boot 3、Spring Security 与 MyBatis-Plus。该技术组合能够满足校园物资管理系统在页面交互、接口组织、权限控制和业务处理等方面的基本需求。

其中，Vue 3 适合构建后台管理类页面，能够较好支持表单、表格、弹窗和统计看板等常见场景；Vite 负责提升前端开发与构建效率；Pinia 用于统一管理登录状态和用户信息；Element Plus 则提供较为完善的基础组件库。后端方面，Spring Boot 3 具有配置简洁、生态成熟和易于分层开发等优点，适合搭建管理信息系统的服务端框架；Spring Security 用于实现登录认证和权限校验；MyBatis-Plus 用于简化常规数据访问操作，提高开发效率。总体来看，该框架组合能够为系统后续功能实现提供稳定的技术基础。

### 2.2 数据存储选型

系统正式运行采用 MySQL 8 作为主要数据库，测试与演示环境采用 H2。之所以选择 MySQL，是因为校园物资管理系统中的用户、仓库、物资、库存、批次、申领单、调拨单和预警记录等数据都具有较强的结构化特征，需要关系型数据库提供较好的事务支持和数据约束能力。

MySQL 在事务处理、索引管理和查询优化方面较为成熟，能够满足系统对库存一致性、业务状态流转和日志留痕的存储需求。H2 则主要用于本地测试和演示环境的快速搭建，便于在不依赖正式数据库的条件下完成自动化测试与功能验证。两者结合，既保证了正式业务数据管理的稳定性，也兼顾了开发阶段的便捷性。

### 2.3 安全与认证技术选型

本系统在安全设计上采用 Spring Security、JWT 和 BCrypt 等技术。由于系统存在管理员、仓库管理员、审批人员和部门用户等多种角色，不同角色对应的页面和接口权限并不一致，因此需要建立较为清晰的认证与授权机制。

Spring Security 主要负责登录认证和接口访问控制，JWT 用于在前后端分离场景下传递用户身份信息，BCrypt 用于对用户密码进行加密存储。与此同时，系统结合 RBAC 权限控制思想，对菜单访问和业务接口访问进行统一约束，以保证不同角色只能执行其职责范围内的操作。该安全技术组合能够较好满足管理系统对身份识别、权限隔离和数据安全的基本要求。

### 2.4 智能分析与可视化技术选型

为了增强系统的数据展示与智能辅助能力，前端选用了 ECharts，后端接入了 DeepSeek 大模型服务。ECharts 主要用于展示库存分布、出入库趋势、部门申领排行和预警统计等内容，使管理人员能够更加直观地掌握业务运行情况。

DeepSeek 主要用于对预警相关数据进行辅助分析，生成风险等级、原因摘要和处置建议等结果，从而为管理人员提供参考。与传统规则预警相比，大模型技术能够在已有业务数据基础上增强信息归纳和风险研判能力，使系统在完成基础管理功能的同时具备一定的智能化分析特征。

### 2.5 本章小结

本章对系统开发过程中采用的主要技术进行了简要说明。前后端分离框架为系统提供了基本开发模式，MySQL 与 H2 满足了正式运行和测试验证的不同需求，Spring Security、JWT 和 BCrypt 为系统安全提供了基础保障，ECharts 与 DeepSeek 则增强了系统的数据可视化与智能分析能力。这些技术共同构成了校园物资智能管理系统的实现基础，并为后续系统设计与实现章节提供了支撑。

## 3 系统可行性分析

### 3.1 技术可行性

从技术角度看，本文设计的校园物资智能管理系统具备较好的实现基础。首先，Vue 3、Spring Boot、MyBatis-Plus、MySQL、JWT 等技术均为成熟方案，适合完成多角色、多页面、多状态流转的管理系统建设。其次，当前仓库已经具备较为完整的模块边界与业务实现，包括认证授权、库存管理、申领审批、调拨处理、预警扫描、统计分析、通知日志和 AI 分析等模块，这说明系统并非停留在概念设计，而是已经形成了可运行、可验证的代码基础。再次，系统使用 H2 支撑测试与截图环境，能够降低联调门槛，提高开发与验证效率。

技术可行性还体现在系统对复杂业务的落地能力上。库存一致性通过事务控制实现，批次处理通过 FEFO 规则实现，申领与调拨通过状态机进行流程约束，预警通过定时扫描与去重逻辑保障结果质量，AI 分析通过结构化输出与回退机制保证稳定性。这些关键机制均已通过服务层、控制层和数据库表结构得到了实际体现。因此，从开发框架、项目实现、模块完整性和验证手段来看，本系统具备良好的技术可行性。

### 3.2 经济可行性

从经济角度看，校园物资智能管理系统采用的技术栈以开源框架为主，不依赖高成本商业中间件，适合毕业设计阶段和高校内部场景。前端框架 Vue 3、后端框架 Spring Boot、数据库 MySQL、图表库 ECharts 以及测试与监控相关的大多数组件都可以在较低成本下部署和维护。对于以校内管理为主的业务环境，系统主要运行在普通应用服务器和关系型数据库环境中即可满足基本需要，不要求额外采购复杂硬件。

此外，系统在设计中兼顾了单机内网部署与后续扩展的可能性。当前版本以单体应用为主，避免了在需求尚不确定时过早引入分布式部署带来的额外成本；同时通过模块化设计、配置化安全参数和标准化 REST API，为后续根据预算与场景扩展能力预留空间。对于高校日常物资管理部门而言，降低初期建设和维护成本，有利于系统从课程设计走向更真实的试用或演示应用。

### 3.3 社会可行性

从社会可行性看，高校物资管理具有明显的公共服务属性。物资管理质量直接影响校园教学、行政和应急保障工作的效率，尤其是在多部门协同和临时需求突增情况下，统一、透明、可追溯的信息系统能够显著降低沟通成本和管理风险。本系统通过规范化物资流转、清晰化审批链路和自动化风险提示，能够帮助管理人员更及时地掌握库存与业务状态，也有利于推动校园内部管理方式从经验驱动向数据驱动转变。

同时，系统在设计中强调日志留痕、角色边界、预警提醒和辅助决策，这些能力符合高校数字治理中对透明度、责任落实和过程管理的要求。随着智慧校园建设不断深入，类似系统在后勤保障、实验室管理、应急物资调度等场景具有较强的借鉴意义，因此在社会应用层面具备一定的可行性和推广价值。

### 3.4 法律可行性

从法律与合规角度看，本系统主要处理校内物资、用户账号、日志和业务单据等管理数据，并不涉及超出校园管理范围的大规模个人敏感信息处理。但即便如此，系统仍需要在账户安全、权限隔离、日志审计和数据使用边界方面保持谨慎。本文在设计中通过密码加密、JWT 鉴权、角色控制、刷新令牌轮换、限流、日志记录和统一异常处理等方式，降低越权访问、重复调用和关键操作缺少追踪的问题。

此外，系统中的 AI 分析功能只用于辅助风险研判，不直接替代人工进行审批、库存扣减或状态变更。也就是说，大模型输出仅作为建议结果存在，最终业务处置仍由具备相应权限的用户完成。这种设计有助于降低自动决策带来的责任不清问题，也更符合当前管理信息系统在审慎使用 AI 技术方面的基本要求。

### 3.5 功能性需求分析

系统的功能性需求主要围绕四类角色展开，即系统管理员、仓库管理员、审批人员和部门用户。不同角色承担不同职责，共同完成物资从建档、入库、申请、审批、出库到签收、调拨、预警与统计分析的完整业务流程。

管理员负责系统基础信息与权限体系维护，包括用户、角色、部门、校区、仓库、库位、供应商、物资分类和物资档案等管理功能。仓库管理员负责库存查询、入库登记、出库处理、仓间调拨执行、预警处理和部分统计查询。审批人员负责对申领单和调拨单进行审批或驳回。部门用户则负责提交申领需求、查看处理进度并完成签收确认。系统通过菜单和接口权限控制，保证不同角色只能访问与自身职责相符的功能页面和业务动作。

在业务功能方面，系统需要至少满足以下几类需求。第一，基础资料管理需求。系统应能维护部门、校区、仓库、库位、供应商、物资分类和物资档案，其中物资安全库存、保质期和所属仓库信息会直接影响预警、补货建议和调拨推荐结果。第二，库存与批次管理需求。系统应能记录入库、出库、汇总库存和批次余量，支持按照效期顺序完成批次选择，并可查询库存状态和批次状态。第三，申领审批需求。系统应支持部门用户创建申领单、提交审批、查询状态，审批人员完成审批，仓库管理员完成出库，最终由部门用户完成签收，形成闭环。第四，调拨管理需求。系统应支持调拨单创建、审批、执行和接收，并在执行时同步修改两侧仓库库存和批次记录。第五，预警与统计需求。系统应能识别低库存、积压、临期、过期和异常出库风险，并提供库存占比、出入库趋势、部门排行、效期统计、仓库分布等分析结果。第六，AI 分析需求。系统应支持对预警记录发起 DeepSeek 分析，并返回结构化处置建议，同时具备失败回退与过程留痕能力。

从业务闭环角度看，本系统并不是若干独立页面的堆叠，而是通过库存、批次、单据、日志和预警记录构建统一的数据链路。例如，申领提交不仅改变单据状态，还会锁定对应库存；出库不仅修改库存，还会回写实际发放数量；调拨执行不仅改变调拨单状态，还会影响源仓与目标仓库存以及批次镜像；AI 分析也不是孤立接口，而是围绕预警记录读取上下文并生成结构化结果。正是这些联动关系，构成了系统的功能完整性。

功能性需求归纳如表 3-1 所示。

| 功能域 | 需求说明 |
|---|---|
| 基础数据管理 | 维护用户、角色、部门、校区、仓库、库位、物资分类、物资档案、供应商和系统配置，为库存、申领、调拨和预警等业务提供基础数据支撑。 |
| 库存与批次管理 | 支持库存分页查询、入库登记、批次建立、批次余量查询、出库扣减和库存盘点；统一维护库存汇总与批次明细数据。 |
| 申领审批与出库 | 部门用户创建并提交申领单，系统锁定可用库存；审批人员完成审批；仓库管理员按申领单执行出库并回写实发数量；部门用户完成签收。 |
| 调拨协同 | 仓库管理员创建调拨单，审批人员审核，仓库管理员执行跨仓搬移；系统按 FEFO 规则扣减来源批次，在目标仓生成镜像批次并同步更新库存汇总。 |

### 3.6 非功能性需求分析

系统的非功能性需求主要包括安全性、一致性、可维护性、可用性、可扩展性和可观测性。

安全性方面，系统需要能够区分未登录、认证失败、账号禁用、权限不足、令牌过期和高频请求等不同情形，并通过统一响应向前端反馈结果。密码必须加密保存，刷新令牌必须具备撤销和轮换能力，高风险接口需要具备限流保护。与此同时，操作日志和登录日志需要记录关键行为，为审计和排查提供依据。

一致性方面，系统的关键风险不在页面展示，而在库存是否准确。入库、出库、申领锁定、锁定释放和调拨执行都会同时影响多张表，因此系统必须在服务层通过事务确保状态流转、库存汇总和批次余量的一致性。若库存扣减只完成一部分或者单据状态提前变化，都会导致后续业务难以追踪，因此一致性是本系统最重要的非功能需求之一。

可维护性方面，系统应具备清晰的模块划分和稳定的接口约定。后端模块按照 `auth`、`rbac`、`inventory`、`apply`、`transfer`、`warning`、`analytics`、`llm` 等职责组织，前端则按页面、路由、组件和接口封装进行管理。分页结构、统一响应结构、业务状态枚举和通用字段设计都应保持尽量一致，便于后续继续扩展与问题定位。

可用性方面，后台管理系统需要在常见桌面浏览器环境中保持稳定显示，页面表单和表格操作要尽量简洁清晰，统计图表要能够直接反映业务重点。用户在完成登录、筛选、查询、审批和处理等动作后，应及时获得成功或失败反馈，避免因为交互不清晰而影响业务处理效率。

可扩展性方面，系统应为未来增加更细粒度权限、更复杂的调拨策略、更多智能分析场景、移动端接入以及更完善的运维能力留出空间。本文当前采用单体应用方式实现，是出于毕业设计阶段复杂度控制的考虑，但模块边界、接口划分和配置方式均为后续演进保留了一定基础。

可观测性方面，系统除了要“能运行”，还要“能观察、能排查”。因此需要暴露基础健康信息与 Prometheus 指标，保留结构化日志，并预留异常追踪能力。对于预警扫描次数、调拨执行次数、登录失败次数等业务行为，系统最好能够形成最基本的监控与记录，以便在演示与维护阶段及时定位问题。

非功能性需求归纳如表 3-2 所示。

| 类别 | 要求 |
|---|---|
| 安全性 | 支持登录认证、JWT 校验、refresh token 轮换、角色菜单与接口授权，并提供限流、安全响应头、登录日志和操作日志。 |
| 一致性 | 库存汇总、批次余量、申领实发数量、库存锁定释放和调拨状态需要在同一事务内协同更新，避免单据与库存数据不一致。 |
| 可维护性 | 后端按模块拆分 Controller、Service、Mapper、Entity，前端按页面、路由、接口封装和通用组件组织，便于后续扩展与维护。 |
| 可测试性 | 提供后端单元测试与集成测试、前端单测、Playwright E2E 测试和 k6 冒烟脚本，用于验证核心流程与接口稳定性。 |
| 可观测性 | 提供 Actuator、Prometheus 指标、JSON 日志和基础观测配置，并支持日志追踪与运行状态监控。 |
| 可用性 | 系统支持 MySQL 正式运行与 H2 测试/截图环境；当 DeepSeek 不可用时，预警分析可回退为规则结果，保证业务连续性。 |

### 3.7 本章小结

本章从技术、经济、社会和法律角度论证了校园物资智能管理系统的可行性，并在此基础上明确了系统的功能性与非功能性需求。分析结果表明，当前项目采用的技术路线成熟可行，成本相对可控，校园应用场景明确，且系统在权限控制、库存一致性、业务闭环和智能辅助分析方面具有清晰的需求边界。这些内容为后续系统设计与实现提供了目标约束和判断标准。

## 4 系统设计

### 4.1 系统架构设计

系统整体采用 B/S 架构和前后端分离设计。用户通过浏览器访问前端管理页面，前端通过 Axios 封装的请求工具与后端 REST API 通信，后端负责认证授权、业务规则处理、事务控制和数据持久化，数据库负责保存业务对象和审计记录。这种架构能够将页面交互与业务逻辑分离，适合校园物资管理中页面较多、状态较多、角色差异明显的业务特点。

从部署视角看，前端可作为静态资源发布，后端作为 Spring Boot 应用运行，正式环境连接 MySQL，测试与截图环境连接 H2。系统同时开放管理端口和业务端口，业务接口主要承载登录、库存、申领、调拨、预警、通知和统计等请求，管理端口用于暴露健康检查与指标信息。系统还预留了 Prometheus、Grafana、Loki 和 Sentry 等可观测性链路，以支撑运行阶段的问题发现和定位。系统总体架构如图 4-1 所示。

从模块边界看，前端主要由登录页、运营总览、库存管理、物资管理、申领审批、调拨管理、预警中心、统计分析、通知中心、日志中心和系统配置等页面构成；后端则按认证、权限、库存、申领、调拨、预警、智能分析、统计、通知和日志等模块拆分。前端负责页面组织、参数封装和结果展示，不承担库存规则和状态流转判断；后端统一落业务状态机、权限边界和事务处理，保证系统行为在接口层面保持一致。

### 4.2 系统关键类设计

为保证系统结构清晰，后端核心类采用分层设计。认证相关的关键类包括 `AuthService`、`JwtTokenProvider`、认证过滤器与安全配置类。其中 `AuthService` 负责登录、刷新、登出和菜单装配，`JwtTokenProvider` 负责 token 生成、校验和解析，安全过滤链负责将认证结果放入上下文中，形成统一的访问入口。与之配套的 `AppStartupValidator` 则负责在非测试环境下校验 JWT 密钥和大模型配置，防止系统以不安全参数启动。系统关键类之间的主要依赖关系如图 4-2 所示。

库存与业务流程相关的关键类包括 `InventoryService`、`ApplyService` 和 `TransferService`。`InventoryService` 负责库存汇总、批次处理、锁定与释放等逻辑，是库存一致性的核心支撑；`ApplyService` 负责申领单的创建、提交、审批、驳回和签收，通过状态变化控制申领流程；`TransferService` 负责调拨单的创建、审批、执行和接收，并在执行时协调源仓与目标仓库存及批次镜像的变化。三个服务共同构成系统主业务链路。

风险与智能分析相关的关键类包括 `WarningService`、`AnalyticsService`、`SmartService` 和 `AiAnalysisService`。`WarningService` 负责定时或手动扫描库存、批次和出库记录，生成不同类型的预警；`AnalyticsService` 负责为前端统计图表提供聚合数据；`SmartService` 负责基于历史出库量生成移动平均预测和补货建议；`AiAnalysisService` 则负责围绕预警记录组装业务上下文、调用 DeepSeek、解析结构化结果并完成回退与留痕。通过这些类的协作，系统智能能力不再是独立模块，而是与预警、统计和业务处理深度耦合。

### 4.3 系统功能设计

#### 4.3.1 用户登录设计

登录设计以“身份可信、权限可控、状态可续签”为目标。用户在前端输入用户名和密码后，由后端进行账号状态和密码校验；校验通过后签发 access token 与 refresh token，并向前端返回用户信息与可访问菜单。前端后续访问受保护接口时统一携带 access token；当 access token 失效时，可通过 refresh token 发起续签，系统校验成功后撤销旧令牌并签发新令牌对。该设计既兼顾了前后端分离场景下的使用便利，也通过轮换和哈希持久化控制了刷新令牌的风险。

#### 4.3.2 基础资料与库存设计

基础资料设计围绕“主数据统一、后续业务复用”展开。部门、校区、仓库、库位、供应商、物资分类和物资档案等对象由管理员统一维护，其他业务模块通过关联主键或基础字段复用这些信息。库存设计则分为汇总库存和批次库存两层：前者描述某种物资在某仓库中的当前数量和锁定数量，后者描述具体批次的剩余量、生产日期和过期日期。入库、出库和调拨等业务都会同时影响这两层数据结构。

#### 4.3.3 申领审批设计

申领审批设计强调闭环和状态约束。部门用户可以先创建草稿单，再发起提交；系统在提交阶段根据申领明细选择可满足全部需求的仓库并锁定库存，随后单据进入待审批状态。审批人员可对待审批单进行通过或驳回处理；通过后由仓库管理员执行出库，系统回写实际发放数量并推进单据至已出库状态；最后由部门用户完成签收，使单据进入已签收状态。若在审批阶段被驳回，则系统释放前期锁定库存。通过这种状态设计，系统将申领、审批、出库和签收组织成完整流程，而非松散动作集合。

#### 4.3.4 调拨管理设计

调拨管理设计主要用于解决多仓库场景下的库存分布不均问题。用户创建调拨单时需明确调出仓、调入仓和物资明细，系统通过草稿、提交、审批、执行和接收等状态表达处理过程。执行阶段是调拨业务的核心：系统需要同时校验来源仓库存、按 FEFO 顺序扣减来源批次、为目标仓生成镜像批次并更新两侧汇总库存。为提高调拨前决策效率，系统还提供候选仓排序接口，在满足需求数量的前提下结合校区距离进行排序，为用户提供参考。

#### 4.3.5 预警与 AI 分析设计

预警与 AI 分析是系统智能能力的核心部分。预警模块负责从库存、批次和出库记录中识别低库存、积压、临期、过期和异常出库等风险，支持定时扫描和手动扫描两种触发方式，并对未处理风险进行去重，避免相同预警持续堆积。统计与智能服务模块则基于历史出库数据、当前库存和安全库存生成补货建议、出库趋势参考和库存汇总图表。

在此基础上，系统新增了面向预警处置的 DeepSeek 分析能力。用户在预警列表中选择具体预警后，系统从物资、仓库、库存、批次、近期出库和待处理业务压力等数据中组装分析上下文，调用 DeepSeek 返回结构化结果，再由后端完成字段归一化与安全过滤。AI 分析结果不直接改变业务状态，而是为管理人员提供风险等级、原因摘要、处置动作、责任角色和时限建议，使智能能力以辅助决策形式嵌入现有流程。为了保证系统稳定性，设计中同时考虑了调用失败时回退规则分析、任务留痕和调用审计等机制。系统功能模块如图 4-3 所示，前后端交互链路如图 4-4 所示，申领审批闭环流程、调拨执行流程和预警处置流程分别如图 4-5、图 4-6 和图 4-7 所示。

#### 4.4.3 索引与存储连接方式设计

## 5 系统实现

### 5.1 系统配置要求

系统开发与运行环境围绕“便于开发、便于测试、便于演示”进行配置。后端使用 Java 17、Spring Boot 3.3.5 和 Maven 构建，前端使用 Node.js、Vue 3 和 Vite 构建。正式数据库为 MySQL 8，测试与截图环境使用 H2。系统通过环境变量配置 JWT 密钥、数据库连接、CORS 参数、限流能力和 DeepSeek 相关参数，使运行环境能够在不修改业务代码的前提下完成切换。

从应用配置看，系统默认开放业务端口用于 API 服务，同时通过独立管理端口暴露健康检查与 Prometheus 指标。认证部分配置包括 access token 有效期、refresh token 有效期、是否允许多设备登录和定时清理策略；限流部分配置包括登录、刷新和高风险操作的容量与时间窗口；DeepSeek 部分则包括启用开关、基础地址、模型名称、超时和最大 token 数等参数。通过配置化方式组织这些关键能力，系统既能适应毕业设计阶段的本地联调，又具备向更规范运行环境迁移的基础。

### 5.2 文件目录层次结构

系统文件结构按照前后端分离原则组织。`backend` 目录保存 Spring Boot 服务端代码、配置文件和测试代码，其中 `modules` 目录按业务领域划分认证、权限、库存、申领、调拨、预警、统计、智能分析、日志和通知等模块；`frontend` 目录保存 Vue 管理端页面、路由、接口封装、状态管理和组件资源；`sql` 目录保存数据库结构与种子数据脚本；`docs` 目录保存架构设计、部署说明、可观测性说明和性能基线记录；`tests` 目录保存性能脚本。该结构有利于将论文中的业务分层与仓库中的实际代码结构对应起来，也便于后续继续维护。

### 5.3 用户基本功能实现

用户基本功能主要包括登录、刷新、登出、当前用户信息获取和菜单装配。登录时，前端将用户名和密码提交到认证接口，后端从用户表读取账号状态，使用 BCrypt 完成密码校验；校验通过后，系统签发 access token 与 refresh token，并把 refresh token 的标识、哈希、过期时间和撤销状态写入持久化表，同时记录登录日志。这样做的好处在于，认证过程不仅返回登录结果，还为后续刷新、审计和安全排查保留了必要依据。

请求鉴权通过 Spring Security 过滤链完成。系统放行登录、刷新和部分公开资源，其余业务接口均要求认证。前端在请求拦截器中携带 access token，后端解析 token 后将当前用户身份放入安全上下文，使业务接口可以基于当前登录人执行权限判断和日志记录。若用户无权访问某接口，系统统一返回权限不足响应，前端据此进行提示或跳转。对于管理员、仓库管理员、审批人员和部门用户，系统会在登录成功后基于角色装配菜单，从页面入口和接口权限两层限制访问边界。登录认证与令牌续签流程如图 5-1 所示，系统登录界面如图 5-2 所示。

refresh token 的实现是用户基本功能中的重点。刷新请求发起后，服务端不仅校验 token 是否过期，还校验其类型、标识、哈希与撤销状态；校验通过后，先撤销旧刷新记录，再签发新 token 对。该流程降低了 refresh token 被重复利用的风险，也使登录态管理更适合前后端分离场景。与此同时，系统还增加了登录与刷新限流，以及启动阶段对 JWT 密钥长度的强校验，从源头减少不安全配置的使用可能。

### 5.4 库存、申领与调拨功能实现

库存业务的核心在于保证“单据变化、汇总库存变化和批次变化保持一致”。入库功能由入库单、入库明细、库存汇总和库存批次共同完成。用户提交入库数据后，系统先创建入库主从单据，再获取或初始化指定仓库中的汇总库存记录，并增加当前数量；随后创建批次记录，写入批次号、入库数量、剩余数量、生产日期和过期日期。由于入库操作会同时涉及多张表，系统将这些步骤放入同一事务中，避免出现单据存在但库存未更新的异常情况。

出库功能在实现上更加复杂。普通出库需要校验汇总库存和批次余量，申领出库则还需要与申领单状态、锁定仓库和申请明细进行关联。系统在出库前优先校验汇总库存是否足够，再按照 `expire_date` 和 `id` 的升序选择可出库批次，也就是采用 FEFO 思路优先扣减更接近失效日期的批次。扣减过程中，系统逐批次减少剩余量，并同步更新汇总库存、锁定数量和出库明细；若是申领出库，还需要把实际发放数量回写到申领单明细，并推进申领单状态。通过这种方式，系统把页面上的一次“出库”动作转化为数据库中的多表联动过程。

申领审批实现体现了系统对业务闭环的重视。部门用户创建申领单时，系统先保存草稿和明细。提交时，系统会汇总申领所需物资，并在库存表中查找能够满足全部需求的仓库，随后锁定相应数量并将仓库编号写入申领单。只有完成库存锁定后，申领单才会进入待审批状态。审批通过时，系统补充审批人和审批意见；审批驳回时，系统释放已锁定的库存；仓库管理员完成出库后，申领单状态进入已出库；部门用户签收后，系统最终释放剩余锁定量并把状态推进到已签收。由于每个状态都对应着库存和单据行为，系统在服务层对状态转换进行了严格限制，避免非法跳转。

调拨实现与申领类似，也以状态控制和事务一致性为核心。调拨单从草稿开始，经由提交、审批、执行和接收等阶段推进。执行调拨时，系统首先校验来源仓库库存是否满足数量要求，然后按 FEFO 顺序扣减来源仓批次，在目标仓生成带有调拨标识的镜像批次，最后同步修改两侧汇总库存。若任一环节出现库存不足或批次不足，整个事务都会回滚，避免出现源仓已扣减而目标仓未增加的中间状态。

为提高调拨前的决策效率，系统实现了候选仓排序接口。该接口首先筛选库存数量满足需求的仓库，再依据固定校区图计算到目标校区的距离，对候选仓进行排序。尽管当前版本没有引入更复杂的运输成本模型，但这一能力已经能在多校区场景下为调拨申请提供明确的候选仓参考，体现出系统从“能处理业务”向“能辅助业务决策”的延伸。部门用户申领界面、库存查询界面和调拨管理界面分别如图 5-3、图 5-4 和图 5-5 所示，出库业务流程图如图 5-6 所示。

### 5.5 预警、AI 分析与统计功能实现

预警功能是系统智能能力的重要入口。`WarningService` 既支持定时任务扫描，也支持页面手动触发扫描。扫描内容包括低库存、库存积压、临期、过期和异常出库五类风险：低库存根据当前库存与安全库存比较得出，积压根据当前库存超过安全库存倍数判断，临期与过期依据批次效期判断，异常出库则根据近 7 天和近 30 天出库量的对比关系识别。每次生成新预警前，系统都会检查是否已存在同类未处理记录，从而避免重复堆积。该逻辑能够让预警记录保持相对简洁，也让真正待处理的风险更突出。

DeepSeek 分析功能围绕预警记录展开。当前端用户点击“AI 分析”后，后端会根据预警 ID 读取相关上下文，包括预警类型、物资档案、安全库存、所在仓库、汇总库存、剩余批次、近 7 天与近 30 天出库量、未完成申领和调拨数量，以及当前未处理预警数量等信息。随后，`AiAnalysisService` 将这些数据组织为请求快照，调用 DeepSeek 生成结构化返回结果。与一般聊天式接入不同，系统要求模型返回固定字段，包括风险等级、摘要、可能原因、建议动作、责任角色和建议时限，从而便于前端直接展示，也便于后端进行校验和归一化处理。

为了保证 AI 接入后的可控性，系统引入了回退和审计机制。若 DeepSeek 未启用、调用失败、返回为空或解析异常，系统会自动构造规则分析结果，确保用户始终能够看到可用建议，不会因为外部模型异常而导致预警处理流程中断。同时，每次分析都会生成任务记录，保存业务对象、请求快照、结果来源和异常信息；模型调用则额外写入调用日志，记录模型名称、提示词模板、token 用量和耗时。这一设计使 AI 分析模块具备工程上的可追踪性，也为后续优化模型调用效果提供了数据基础。

统计与智能推荐功能由 `AnalyticsService` 和 `SmartService` 提供支持。统计分析模块对库存占比、出入库趋势、部门排行、效期统计、仓库分布和应急消耗等主题进行聚合，并通过前端图表展示给管理人员。补货建议模块结合安全库存、当前库存、近 30 天出库量和保障天数计算目标库存与建议补货量，帮助仓库管理员掌握近期需要优先补货的物资。趋势分析模块则以近 6 个月历史出库数据为基础计算移动平均值，生成未来月份的参考数量。虽然这些能力并不直接驱动库存变化，但它们为管理人员提供了更全面的观察视角，使系统的“智能化”不再局限于单点预警，而是覆盖预警识别、原因分析和决策支持。预警管理界面、运营总览界面和统计分析界面分别如图 5-7、图 5-8 和图 5-9 所示，统计数据流转流程如图 5-10 所示。

### 5.6 其他功能实现

通知与日志功能为系统提供了过程留痕和消息提醒能力。`notification` 表保存审批通知、预警提醒和业务事件消息，前端支持列表查询、未读数查看、已读标记、全部已读和删除等操作。登录日志由认证流程自动写入，记录用户、IP、登录状态、登录时间和 user agent 等信息；操作日志则在创建、提交、审批、出入库、调拨、预警处理和 AI 分析等关键动作后写入，便于后续追踪谁在何时对哪个模块执行了什么操作。

在工程化实现方面，系统还补充了分页治理、限流、安全响应头和可观测性能力。针对库存、预警、操作日志等高增长列表，系统统一采用后端分页模式，避免前端全量拉取导致性能下降；针对登录、刷新和高风险操作，系统增加了限流过滤器；响应层增加了常见安全头，以增强浏览器端安全边界；运行层则开放 Actuator 和 Prometheus 指标，并结合结构化日志与 Loki、Grafana、Sentry 等方案构成最小可观测体系。尽管这些能力不直接体现为单个业务页面，但它们共同提升了系统的稳定性、可维护性和可交付程度。

### 6.5 本章小结

通过后端单元测试、后端集成测试、前端单元测试、Playwright 端到端测试以及 k6 本地性能基线测试，可以看出本系统在认证授权、库存管理、申领审批、调拨执行、预警处理、通知提醒和 AI 分析等方面已经具备较完整的可验证基础。测试结果表明，系统核心业务闭环能够按预期推进，接口在异常参数、权限不足和高风险操作等场景下也能给出清晰反馈。同时，分页接口和登录接口已经具备可重复比较的本地性能基线，为后续优化与回归提供了参考依据。

## 结束语

本文围绕高校物资管理中信息分散、流程不透明、库存批次追踪困难和风险处置效率不足等问题，设计并实现了一套基于 B/S 架构的校园物资智能管理系统。系统以 Spring Boot 和 Vue 为技术基础，将认证授权、库存与批次管理、申领审批、出库签收、仓间调拨、预警识别、通知日志和统计分析等能力整合为统一平台，并通过事务控制、状态约束、分页治理和审计留痕等手段保证核心业务链路的稳定性与一致性。

与传统仅强调规则提醒的管理系统相比，本文进一步将 DeepSeek 大模型分析能力嵌入预警处置流程中，使系统能够围绕真实业务上下文生成结构化风险说明与处置建议，并通过任务表、调用日志和回退机制保证分析过程可追踪、可回溯、可降级。由此形成的“预警识别—原因分析—建议生成—人工处置”链路，使校园物资管理系统在信息化基础上进一步具备了辅助决策支持能力。

当然，本文工作仍然存在一定局限。当前系统主要面向单机内网与毕业设计演示场景，尚未扩展到更复杂的分布式部署环境；AI 分析功能目前聚焦预警场景，尚未扩展到调拨说明生成、统计报告总结等更多业务方向；性能测试主要基于本地基线结果，后续仍需要在更贴近真实运行环境的条件下继续验证。下一步可围绕更细粒度权限配置、移动端适配、调拨策略细化、模型提示词优化、智能报告生成和可观测性完善等方向继续扩展。

总体来看，本文完成的校园物资智能管理系统既覆盖了物资管理的核心业务流程，也在工程化与智能化方面进行了较为完整的探索，能够为类似校园后勤、实验室和应急保障场景下的物资管理平台建设提供一定参考。
